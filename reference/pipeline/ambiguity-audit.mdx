## Purpose

This is the most important anti-hallucination stage in the entire pipeline. Before generating a single line of code, you systematically audit every implementation-relevant detail and classify it as `SPECIFIED`, `PARTIALLY_SPECIFIED`, or `UNSPECIFIED`. This stage produces the raw material for `REPRODUCTION_NOTES.md`.

**Assume nothing. Verify everything.**

When you read "we use standard settings," ask: standard according to whom? When you read "following prior work," ask: which prior work, and which specific detail from it? When something seems obvious, ask: is it obvious because it's specified, or because you're making an assumption?

The goal is a document where any reader can look at any implementation choice and immediately know whether the paper specified it or whether you chose it.

## Input

- Parsed paper sections from Stage 1
- Contribution analysis from Stage 2 (includes official code status)
- Official code repository (if found in Stage 1 — URL in `paper_metadata.json` under `official_code`)

## Output

- `.paper2code_work/{ARXIV_ID}/ambiguity_audit.md` — complete audit with classification for every item

## The complete checklist

For each item below, search the paper (method section, experiments section, appendix, footnotes — all of them) and classify it as one of:

- **`SPECIFIED`** — the paper states this explicitly. Record the exact quote and section.
- **`PARTIALLY_SPECIFIED`** — the paper mentions it but is ambiguous. Record the quote and what is unclear.
- **`UNSPECIFIED`** — the paper does not state this at all. Record common choices and which one you will use as a default.

## How to search the paper

For each checklist item, search in this order:

1. **Method section** — the obvious place
2. **Experiments section** — often contains an "Implementation Details" or "Training Details" subsection
3. **Appendix** — the most common hiding spot for training details
4. **Figure captions** — sometimes specify architectural details not present in the text
5. **Footnotes** — authors sometimes put caveats and clarifications in footnotes
6. **Table captions** — hyperparameter tables often have footnotes with additional context
7. **Keyword search** — search the paper text for "learning rate", "dropout", "optimizer", etc.

## How to handle "following X" references

When the paper says "following [X]", "similar to [X]", or "as in [X]":

1. Note the reference — what is paper X?
2. If X is a well-known paper (e.g., "following Vaswani et al."), describe what X specifies
3. If X is an obscure paper, flag it as requiring the reader to look up that paper

Record it as:

```
[PARTIALLY_SPECIFIED] Paper says "following [X]" for {component} — X specifies {detail} but reader should verify
```

## How to resolve ambiguities using official code

If official code was found in Stage 1, use it as the primary resource for resolving `UNSPECIFIED` and `PARTIALLY_SPECIFIED` items.

**Process:**

1. For each `UNSPECIFIED` item, search the official repo for the relevant detail (e.g., look for `eps`, `dropout`, `lr` in config files and model code)
2. If you find the answer, update the classification:
   - `UNSPECIFIED` → `SPECIFIED` with tag `[FROM_OFFICIAL_CODE]`
   - `PARTIALLY_SPECIFIED` → `SPECIFIED` with tag `[FROM_OFFICIAL_CODE]`
3. Record the exact file and line from the official repo: `github.com/author/repo/blob/main/model.py#L42`

**What official code can resolve:**
- Activation functions, normalization epsilon, dropout placement
- Initialization schemes — often the biggest silent assumption
- Learning rate schedule details, optimizer parameters
- Data preprocessing steps
- Any "standard settings" or "following prior work" references

**What official code cannot resolve:**
- Whether the code matches the paper's intent (bugs exist in official code too)
- Whether a detail was chosen for the paper's experiments vs. for engineering convenience

<Warning>
  Official code is a reference, not ground truth. If the official code contradicts the paper, note both and flag the discrepancy rather than silently following one or the other.
</Warning>

## How to handle errata and contradictions

**Figure vs text disagreement:** Flag it explicitly. Implement what the text and equations say — they are usually more carefully reviewed than figures — but note that the figure shows something different.

**Abstract vs method section disagreement:** The method section wins. The abstract is a summary and may oversimplify.

**Equation vs prose disagreement:** The equation wins. It is more precise. Flag the discrepancy.

**Paper vs official code disagreement:** Note both. The official code may fix bugs or reflect later understanding, but the paper is what was peer-reviewed. Record it as:

```
Paper says X in §Y.Z, but official code does W — we implement {choice} because {reason}
```
