paper2code is a skill for Claude Code that converts arXiv research papers into working Python implementations. Unlike naive code generation, it audits every implementation detail before writing a single line — classifying each choice as specified, partially specified, or unspecified. The result is code you can trust because you can verify every decision against the paper.

## Why paper2code

ML papers are notoriously vague. Critical hyperparameters are buried in appendices or omitted entirely. "Standard settings" refers to nothing specific. When you implement a paper manually, you spend more time detective-working than coding.

Naive LLM code generation makes this worse — it fills every gap silently and confidently. You get something that runs but doesn't match the paper, and you can't tell which parts are from the paper and which were invented by the model.

paper2code solves this with a 5-stage pipeline:

<Steps>
  <Step title="Paper acquisition">
    Fetches the arXiv PDF and extracts full text with math notation preserved. Appendices, footnotes, and figure captions are treated as first-class sources.
  </Step>
  <Step title="Contribution identification">
    Identifies the single core contribution of the paper and classifies the paper type to determine what to implement.
  </Step>
  <Step title="Ambiguity audit">
    Every implementation-relevant detail — architecture, training, data, evaluation — is classified as SPECIFIED, PARTIALLY_SPECIFIED, or UNSPECIFIED before any code is written.
  </Step>
  <Step title="Code generation">
    Generates citation-anchored Python code. Every class references the paper section it implements. Every unspecified choice is flagged with alternatives.
  </Step>
  <Step title="Walkthrough notebook">
    Creates a Jupyter notebook linking paper sections to code with runnable sanity checks on CPU with toy dimensions.
  </Step>
</Steps>

## Key features

<CardGroup cols={2}>
  <Card title="Citation anchoring" icon="link">
    Every non-trivial code decision is anchored to the paper with `§3.2` or `§3.2, Eq. 4` style references inline.
  </Card>
  <Card title="Honest uncertainty" icon="circle-question">
    Unspecified choices are flagged with `[UNSPECIFIED]` comments listing the chosen default and alternatives.
  </Card>
  <Card title="Appendix mining" icon="magnifying-glass">
    Appendices, footnotes, and figure captions are treated as first-class specification sources — not ignored.
  </Card>
  <Card title="Official code lookup" icon="code-branch">
    Automatically searches for and cross-references the authors' official implementation to resolve ambiguities.
  </Card>
  <Card title="Multiple modes" icon="sliders">
    Minimal (default), full (training + data pipeline), and educational (extra comments + paper guide) modes.
  </Card>
  <Card title="Framework choice" icon="layer-group">
    Generate implementations in PyTorch (default), JAX, or NumPy.
  </Card>
</CardGroup>
